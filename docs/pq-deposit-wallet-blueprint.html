PQ Deposit (ML-DSA-65) wallet create/withdraw — implementation blueprint

Generated by the design workflow (4 parallel code-readers + synthesis). Spec for implementing pq_deposit/pq_withdraw under Option 3. All file:line refs verified against pqc/testnet-poc at the time of writing.

PQ Deposit (ML-DSA-65) — Wallet Create/Withdraw Implementation Blueprint

Scope: under Option 3 (classical deposit creation frozen at V9, per Blockchain.cpp:3121-3158), build the wallet-side CREATE + WITHDRAW path for PqMultisigOutput/PqMultisigInput. Consensus, interest, lock, and double-spend (isUsed) are already done daemon-side (Blockchain.cpp:3799-3912, Currency.cpp:444-468). All remaining work is wallet-side + one new daemon enumeration RPC. Testnet-only until V9 mainnet audit (UPGRADE_HEIGHT_V9=5000000 sentinel).

1. New wallet surface

concealwallet (CLI) — mirror deposit/withdraw (ConcealWallet.cpp:344,1672-1768), wire like pq_transfer (:349-352,2151-2266)

pq_deposit <months> <amount>      // create a PQ deposit
pq_withdraw <deposit_id>          // withdraw a matured PQ deposit
  • pq_deposit: parse months → term = months * m_currency.depositMinTermV3() (testnet 30, mainnet 21900; Currency.h:476-478, CryptoNoteConfig.h:73,134). Validate depositMinTermV3 <= term <= depositMaxTermV3 && term % depositMinTermV3 == 0 and amount >= depositMinAmount — byte-identical to classical so the wallet pre-validates exactly what Currency::validateOutput enforces (Currency.cpp:1419-1443). Resolve the wallet's own PQ ML-DSA keys via a new getPqAccountKeys()-style call (currently getPqAccountKeys() derives only KEM — see §3). Route to the new builder/relay helper (§2) with m_dispatcher/m_daemon_host/m_daemon_port, exactly as pq_transfer calls pqSpendViaDaemon (:2225-2249).
  • pq_withdraw: parse one uint64 id, look it up in the wallet's PQ-deposit table (§3 list), require it matured (createdHeight + term <= currentHeight) and not already spent, build+sign+relay the withdrawal.
  • Register both in the command table next to pq_balance/pq_transfer/pq_receive (:349-352).

walletd (RPC) — mirror createDeposit/withdrawDeposit (WalletService.cpp:1559-1638) wired like sendPqTransaction (:1232-1267)

createPqDeposit   { sourceAddress?, amount, term }  -> { transactionHash }
withdrawPqDeposit { depositId }                     -> { transactionHash }
  • Register handlers in PaymentServiceJsonRpcServer.cpp (mirror :58,217-218 for sendPqTransaction; the classical pair is at :66,68,246,251).
  • Add CreatePqDeposit/WithdrawPqDeposit Request/Response in PaymentServiceJsonRpcMessages.{h,cpp} (mirror SendPqTransaction, :523).
  • Reuse the classical validation guards: daemon-synced, term % depositMinTermV3()==0, term band, amount >= depositMinAmount (WalletService.cpp:1559-1600).

2. Construction path

Decision: add a new shared builder buildPqDepositTransaction + buildPqWithdrawTransaction in src/CryptoNoteCore/ (mirror PqSpendBuilder), NOT a method bolted onto WalletGreen::createDeposit. Rationale: PqSpendBuilder is already the single front-end-agnostic, pure (no I/O/globals) PQ tx constructor that concealwallet, walletd, and pq_injector all share (PqSpendBuilder.h:34-60). Keeping the PQ multisig path in the same layer means one verified construction site, consistent with the existing PoC architecture, and avoids forking logic across WalletGreen/WalletService.

Prerequisite P0 — ITransaction PQ writer API (HARD BLOCKER)

ITransactionWriter (ITransaction.h:108-124) has no PQ methods. TransactionImpl (Transaction.cpp:303-401) implements only Ed25519 MultisignatureOutput/Input + signInputMultisignature. Two construction strategies:

  • Strategy A (preferred): bypass ITransaction, assemble cn::Transaction directly — exactly what PqSpendBuilder already does (PqSpendBuilder.cpp:171-227 sets tx.version, tx.inputs, tx.outputs on a raw Transaction). This sidesteps the missing ITransaction virtuals entirely and matches the proven PQ PoC pattern. Use this.
  • Strategy B (avoid): extend ITransaction with addOutput(amount, pqKeys, requiredSigs, term) / addInput(PqMultisigInput) / signInputPqMultisignature. More invasive, touches the legacy interface, not needed if A is used.

CREATE — buildPqDepositTransaction(req, tx, err)

Mirror classical createDeposit shape (WalletGreen.cpp:447-574) but emit a PqMultisigOutput instead of calling addOutput(...,term):

  1. tx.version = TRANSACTION_VERSION_3 (required by the visitor, Blockchain.h:513-533), unlockTime = 0.
  2. Funding input: select normal funds for amount + fee (fee=1000) and add as KeyInput(s) with mixin — reuse WalletGreen's selectTransfers/prepareInputs/signInputKey (:489-568). (Alternatively a PqKeyInput ring-spend, but classical-funded is simplest and matches classical_deposit_injector.cpp:157-170 shape.)
  3. Deposit output = PqMultisigOutput{ keys = [walletDsaPubKey] (n=1), requiredSignatureCount = 1 (m=1), term }, amount = neededMoney - fee. Each key MUST be ccx_pq_multisig_pubkey_bytes()=1952B (CryptoNoteFormatUtils.cpp:435-456), keys.size() <= PQ_MULTISIG_MAX_KEYS=16, requiredSignatureCount ∈ [1, keys.size()].
  4. Change outputs → source address, split/shuffled (as :511-535).
  5. Sign the KeyInputs; relay.

n-of-m note: for a single-wallet product, n=m=1. The struct supports n-of-m, but multi-party key exchange is out of scope; document as future.

WITHDRAW — buildPqWithdrawTransaction(req, tx, err)

Mirror classical withdrawDeposit (WalletGreen.cpp:313-392) + prepareMultisignatureInputs (:425-445) but build a PqMultisigInput:

  1. tx.version = TRANSACTION_VERSION_3.
  2. Input = PqMultisigInput{ amount, signatureCount = output.requiredSignatureCount, outputIndex (global index into m_pqMultisigOutputs[amount]), term = output.term, signatures = EMPTY }. term == output.term is a hard consensus bind (Blockchain.cpp:3857).
  3. Output(s) = pay principal + interest back to the wallet (PqKeyOutput or normal KeyOutput). Do NOT compute interest in the wallet for consensus — see §3/§5: the daemon credits amount + getInterestForInput and allows outputs > inputs by exactly that. The wallet must set the payout to match the consensus value or the tx fails balance. Safest: payout = amount + currency.calculateInterest(amount, term, createdHeight) and rely on parity (the wallet's calculateInterest mirrors the daemon's getInterestForInput; confirm parity in test — §6).
  4. Compute signing hash (§4), sign, fill signatures inline.

Relay — reuse the PqSpendClient pattern (PqSpendClient.cpp:370-386)

rreq.tx_as_hex = common::toHex(cn::toBinaryArray(tx));
invokeJsonCommand(httpClient, "/sendrawtransaction", rreq, rres);  // check rres.status == CORE_RPC_STATUS_OK
return common::podToHex(cn::getObjectHash(tx));

Add a new Rpc/PqDepositClient.{h,cpp} helper (analogous to PqSpendClient::pqSpendViaDaemon) exposing pqDepositViaDaemon(...) and pqWithdrawViaDaemon(...).

3. KEY strategy (deterministic-from-seed, mirroring the KEM precedent)

Decision: derive the deposit ML-DSA-65 keypair DETERMINISTICALLY from the wallet master seed via a new domain tag — AND persist the public-key-to-deposit mapping minimally. Do NOT store the secret key per-deposit. Rationale (grounded in facts):

  • The deterministic FFI already exists and is proven interoperable: ccx_pq_multisig_keygen_det(seed,...) (pq_ring_sig.h:47,77, detkeygen.rs:120-136), domain ccx-pq-mldsa65-keygen-v1, deterministic + pqcrypto-interop selftest (detkeygen.rs:210-212). No missing primitive — this is NOT a blocker (corrects the prompt's "missing deterministic ML-DSA keygen FFI" worry; it is present).
  • The established wallet precedent is derive-from-seed at enable time, persist into the versioned v8 PQ section, restorable both ways (WalletGreen.cpp:1399-1413,1329-1397). Follow it.

Prerequisite P1 — add ML-DSA derivation to PqAccount (PqAccount.cpp:38-64, PqAccount.h:28-34):

  • Add dsaPublicKey/dsaSecretKey/dsaSchemeId to PqAccountKeys.
  • Add deriveDsaSeed(master) = cn_fast_hash("ccx-pq-multisig-acct" || master32) — same pattern as deriveKemSeed (:28-36), distinct domain tag (KEM/ring/DSA seeds independent, per detkeygen.rs:32-35).
  • In generateFromSeed, also call ccx_pq_multisig_keygen_det(dsaSeed, ...) → pk=1952/sk=4032.

The hard sub-problem — ML-DSA has no stealth derivation. Classical deposits re-derive a per-output stealth key from (txPubKey, outputIndex, accountKeys) (Transaction.cpp:303-323,368-391); ML-DSA has no derive_public_key analogue (deposit-key-storage gaps). Two viable designs:

  • D1 (recommended for v1): single account-level ML-DSA keypair. Every PQ deposit's PqMultisigOutput.keys[0] = the wallet's one account DSA pubkey. Withdraw = sign with the one account DSA secret. Pro: fully seed-restorable with zero per-deposit state; the wallet recognizes its deposits by matching the on-chain output key against its own account DSA pubkey (see scanner, P2). Con: deposits to the same wallet are linkable by the shared pubkey (privacy regression vs. classical stealth). Acceptable for a testnet PoC; flag the privacy note.
  • D2 (future): per-deposit indexed keys. deriveDsaSeed(master || counter) with a monotonically increasing index persisted in the v8 PQ section. Restores by re-deriving indices 0..N and matching pubkeys. More privacy, but needs an index counter + a gap-limit scan on restore. Defer.

Pick D1. Restore works purely from the mnemonic (re-derive the one account DSA keypair); the v8 PQ section bump is optional (store the DSA pubkey for fast match, but it's re-derivable). If you do persist, bump pqSectionVersion to 2 and append [dsaSchemeId][dsaPublicKey][dsaSecretKey] after the KEM fields (WalletGreen.cpp:1356-1396).

4. Withdraw signing — the message is the PQ signing hash, NOT a plain object hash (CONFIRMED)

This is the single most error-prone detail. Confirmed from Blockchain.cpp:2603-2621,2453,3900-3903:

  • The validator computes getTransactionPqSigningHash(tx) = hash of the TransactionPrefix after clearing every PqKeyInput.ringSig AND every PqMultisigInput.signatures (a sig cannot commit to itself). check_pq_multisig verifies each detached sig over this exact hash (Blockchain.cpp:3900-3903).
  • Builder contract: assemble the full prefix with PqMultisigInput.signatures empty, compute the hash by the same clearing rule, have each of the m signers call ccx_pq_multisig_sign(hash, sk, ...) (pq_ring_sig.h:74-83), then write the m sigs inline into PqMultisigInput.signatures.
  • Ordering: check_pq_multisig does a greedy m-of-n in-order match — each input sig must ccx_pq_multisig_verify against a distinct, in-order output key (Blockchain.cpp:3799-3912). For n=m=1 trivial; for n-of-m, place sigs in the same order as the output keys they correspond to.
  • Lock/maturity: if output.term != 0, consensus requires cell_block_height + output.term <= current_height (Blockchain.cpp:3857-3869). The wallet must surface spendableHeight = createdHeight + term and block pq_withdraw before it (mirror classical withdrawDeposit's unlockHeight <= blockCount check, WalletGreen.cpp:313-392).
  • getSignaturesCount(PqMultisigInput) == 0 (sigs are inline, not in tx.signatures) — so leave tx.signatures empty for the PQ input (include/CryptoNote.h:60-66).

5. Open risks + MUST-validate

  1. Interest is daemon-computed, not wallet-computed (consensus). Currency::calculateTotalTransactionInterest sums getInterestForInput per PqMultisigInput with term!=0 and credits already_generated_coins (Currency.cpp:444-468, Blockchain.cpp:802,849,3188,3211,3218,2482). The withdraw tx is allowed outputs > inputs by exactly this interest. MUST validate: the wallet's payout amount equals amount + daemon-interest, else the tx is rejected (imbalance) or burns interest. Verify wallet calculateInterest == daemon getInterestForInput for the chosen term/height in a unit test.
  2. Double-spend / isUsed. Spend marks m_pqMultisigOutputs[amount][outputIndex].isUsed=true, serialized to block cache, reorg-symmetric (Blockchain.cpp:773,794-798,3601-3646). MUST validate: a second pq_withdraw of the same cell is rejected; wallet should not offer an already-spent deposit.
  3. Term band / % depositMinTermV3. Output acceptance enforces the term band byte-for-byte with Ed25519 (Currency.cpp:1419-1443). MUST validate wallet pre-check matches; off-band terms must be rejected client-side with a clear message.
  4. input.term == output.term bind (interest-minting safety, Blockchain.cpp:3857). The withdraw builder MUST copy term from the tracked output (mirror prepareMultisignatureInputs, WalletGreen.cpp:425-445).
  5. The freeze does NOT block PQ deposits. At/after V9, classical MultisignatureOutput with term!=0 is rejected but PqMultisigOutput is the only allowed new deposit (Blockchain.cpp:3121-3158). MUST validate: post-V9, classical deposit rejected AND pq_deposit accepted (this is the core Option-3 acceptance test).
  6. Scanner blindness (P2 below). Until TransfersConsumer classifies PqMultisigOutput, the wallet cannot see/list/mature/withdraw its own deposit — the whole feature is non-functional without it.
  7. Key well-formedness at FFI boundary: every output key length re-checked == 1952 in check_outs_valid (CryptoNoteFormatUtils.cpp:435-456) and again at verify. The builder must use the exact ccx_pq_multisig_pubkey_bytes() length.
  8. Unaudited / non-constant-time PoC. ML-DSA integration flagged unaudited; mainnet UPGRADE_HEIGHT_V9 is a sentinel. Ship testnet-only.

Prerequisites (must land before the feature works)

  • P0 — tx construction surface. Use Strategy A (assemble raw cn::Transaction à la PqSpendBuilder); no ITransaction change needed. (If Strategy B chosen instead, add 3 virtuals to ITransaction.h:108-124 + Transaction.cpp impls.)
  • P1 — PqAccount ML-DSA derivation (ccx_pq_multisig_keygen_det, domain ccx-pq-multisig-acct). FFI exists; only the C++ wiring is missing.
  • P2 — TransfersConsumer/Container PQ deposit scanning (HARD BLOCKER). TransfersConsumer.cpp:87 and :450 only handle OutputType::Multisignature; deposit classification at TransfersContainer.cpp:960-961 keys off Multisignature && term>0. A PqMultisigOutput branch must: detect ownership (match keys[0] against the wallet's account DSA pubkey — there is no KEM/stealth scan for multisig, it's a named-key match), surface it as a deposit output with term, globalOutputIndex, createdHeight, so the existing m_deposits machinery (insertNewDeposit/insertDeposit/getDeposit/getDepositId, WalletGreen.cpp:4037-4091) works. Note insertNewDeposit asserts type==Multisignature (:4037-4057) — must add a PQ branch or a parallel m_pqDeposits table. This is the largest sub-task.
  • P3 — daemon enumeration RPC get_pq_multisig_outputs (or get_pq_deposits). get_pq_outputs/Core::getPqOutputs only walk m_pqOutputs (ring), not m_pqMultisigOutputs (RpcServer.cpp:924-968, Core.h:80). Needed for the withdraw builder to resolve (amount, outputIndex, term, height, isUsed) of the owned cell. Mirror the get_pq_outputs RPC shape with DoS caps. (If P2 tracks the cell's global index fully wallet-side, P3 may be reducible to a confirm-not-spent check via existing RPC — but the safe path is a dedicated enumeration.)

6. Phased build order + test plan

Phase 1 — keys (P1). Add ML-DSA fields + deriveDsaSeed + generateFromSeed DSA branch to PqAccount. Optionally bump v8 PQ section to v2.

  • Unit: determinism (same master → same DSA pk/sk); pk=1952/sk=4032; ccx_pq_multisig_keygen_det interop; KEM/DSA/ring seeds distinct. Extend the existing determinism asserts referenced in PqAccount.h:13.

Phase 2 — builders (P0 Strategy A). buildPqDepositTransaction + buildPqWithdrawTransaction in CryptoNoteCore, pure functions like PqSpendBuilder.

  • Unit (new TestPqDepositTx.cpp, going beyond TestPqDeposits.cpp:48-298 which stops at primitives): build a real PqMultisigOutput deposit tx and a PqMultisigInput withdraw tx; assert the withdraw signing hash = getTransactionPqSigningHash (signatures cleared); run both through a stood-up Blockchain/check_pq_multisig (the existing test never does this) and assert ACCEPTED; assert wrong-message sig → REJECTED; assert input.term != output.term → REJECTED; assert interest parity (wallet calculateInterest == consensus credit).

Phase 3 — scanner (P2). PQ deposit detection in TransfersConsumer/Container + m_deposits/m_pqDeposits integration.

  • Integration: after a PQ deposit confirms, getWalletDepositCount()/list shows it with correct amount/term/spendableHeight; survives wallet reload (re-derive DSA key, re-match).

Phase 4 — enumeration RPC (P3). get_pq_multisig_outputs + Core/ICore method.

  • Integration: RPC returns the owned cell's (amount, outputIndex, term, height, isUsed).

Phase 5 — relay helpers + front-ends. PqDepositClient, then concealwallet pq_deposit/pq_withdraw and walletd createPqDeposit/withdrawPqDeposit.

  • Integration: CLI end-to-end against testnet daemon.

Phase 6 — e2e (extend pqc/verify-deposit-freeze.sh). Mine past TESTNET_UPGRADE_HEIGHT_V9=80, then:

  1. classical deposit → REJECTED (freeze proven).
  2. pq_deposit <months> <amount> → ACCEPTED, confirmed, visible in wallet.
  3. mine term blocks → deposit matures (createdHeight + term <= height).
  4. pq_withdraw <id> → ACCEPTED; principal+interest credited; isUsed set.
  5. second pq_withdraw <id> → REJECTED (double-spend).
  6. reload wallet from mnemonic before step 4 → deposit still withdrawable (restore proven).
